论文阅读-Declarative Smart Contracts

发表于2022-11-18|更新于2022-11-18|paper-reading
|字数总计:1.4k|阅读时长:5分钟|阅读量:

总结

摘要：This paper presents DeCon, a declarative programming language for implementing smart contracts and specifying contract-level properties. Driven by the observation that smart contract operations and contract-level properties can be naturally expressed as relational constraints, DeCon models each smart contract as a set of relational tables that store transaction records. This relational representation of smart contracts enables convenient specification of contract properties, facilitates run-time monitoring of potential property violations, and brings clarity to contract debugging via data provenance. Specifically, a DeCon program consists of a set of declarative rules and violation query rules over the relational representation, describing the smart contract implementation and contract-level properties, respectively. We have developed a tool that can compile DeCon programs into executable Solidity programs, with instrumentation for run-time property monitoring. Our case studies demonstrate that DeCon can implement realistic smart contracts such as ERC20 and ERC721 digital tokens. Our evaluation results reveal the marginal overhead of DeCon compared to the open-source reference implementation, incurring 14% median gas overhead for execution, and another 16% median gas overhead for run-time verification.

来源：ESEC/FSE 2022 https://dl.acm.org/doi/abs/10.1145/3540250.3549121

概览

这个问题并不是很重要，而是提出了一种比较新奇的方式去结合了 Datalog 和 Solidity，意义在于创新性很好。总的来说，提出了基于 LP 的 domain specific language。在此之前，已经有人提出来过根据不变量去生产断言，然后用于运行时验证了。Azzopardi et al. 也提出了对比合约逻辑的检测检测方法。另外，

Scilla [56] is a intermediate-level language for smart contracts that offers type safety and support for verification. KEVM [36] defines the formal semantics of EVM, and has been used to verify contracts against ERC20 standards.

Jones et al. [38] uses functional programming language to write financial contracts.

BitML [20] is a high-level language for Bitcoin smart contracts. Based on process calculus, it translates contracts into Bitcoin transactions.

以上文章给他们设计 specification 带来了灵感。另外还需要少部分数理逻辑内容才能完成 domain specific language 设计。

设计

基本和 souffle 一致，不赘述了，如果读者不明白可以看往期 Datalog 的文章，善用博客的搜索功能。下面说一些特有的设计。

  • 所有 declare 都是关系表，并且可以用下标指定主键，如果没有下标，默认全部主键。
  • 保留了一些函数，用于表示合约特性，比如 msg 之类的。
  • 加入了特殊的 declaration 用于表示异常情况。执行过程之后，如果 violation 还存在，那么就回退交易。
  • 加入了 public 标签，用于直接查询表，类似于 output
  • recv_ 前缀的 declaration 用于当作触发器，是执行的起点。

局限性和亮点

局限性：

  1. Keecak 这些密码学函数和 ABI 很难放在 logic language 去实现。至少作者按照 souffle 的逻辑是说很难。
  2. 继承关系、接口、函数调用时的函数签名匹配这些，也没做好来。简单地说，好做的就做了，不好做的先放着。
  3. 根据推导规则回溯去寻找变量更改的源头，但是可能存在多个分支，要么是无法精确定位，要么是指数级复杂度。
  4. declaration 都用关系表表示，solidity 中是映射+结构体，实际上这种生成出来的代码效率很低的。

亮点：

  1. 作者的编程语言基础挺好的，值得学习。
  2. LP 转 solidity 是个不错的设想，创新性好，值得鼓励。

论文阅读-A Study of Inline Assembly in Solidity Smart Contracts

发表于2022-11-01|更新于2022-11-01|paper-reading
|字数总计:1.6k|阅读时长:7分钟|阅读量:

摘要

This paper presents a large-scale quantitative study of the use of inline assembly in 6.8𝑀 smart contracts deployed on the Ethereum blockchain. We find that 23% of the analyzed smart contracts contain inline assembly code, and that the use of inline assembly has become more widespread over time. We further performed a manual qualitative analysis for identifying usage patterns of inline assembly in Solidity smart contracts. Our findings are intended to help practitioners understand when they should use inline assembly and guide developers of Solidity tools in prioritizing which parts of inline assembly to implement first. Finally, the insights of this study could be used to enhance the Solidity language, improve the Solidity compiler, and to open up new research directions by driving future researchers to build appropriate methods and techniques for replacing inline assembly in Solidity programs when there is no real necessity to use it.

来源：顶会-OOPSLA2, Article 165 (October 2022), 27 pages. https://doi.org/10. 1145/3563328

具体设计

  1. Data Collection.
    1. query Google Big Query (similar to Durieux et al. [2020]) to get the addresses of contracts deployed on Ethereum and retrieve specific metadata.
    2. excludes contracts that have never been used, i.e., they have not any transaction or token transfer.
    3. use Etherscan and its API to retrieve the source code associated with an address.
    4. proceed to (1) parse the output of Etherscan and save each Solidity contract in a single file in the filesystem, 4 and (2) detect possible duplicates.(first remove all spaces and tabulations from contracts, and then compare their SHA256 checksums)
  2. Quantitative Analysis
    1. analyzing Solidity smart contracts using a customized parser built on top of solidity_parse ,
    2. feeding the results to a database constituted by the following tables: “NonAssemblyContractAddress”, “AssemblyContractAddress”, “Label”, “AddressLabel”, “SolidityFile”, “Contract” (here, we refer to contracts as we would refer to classes in Java), “InlineAssemblyFragment”, “Instruction”, and “InstructionsPerFragment” , and
    3. executing queries to get quantitative results about inline assembly usage.

  1. Qualitative Analysis
    1. we first cluster contracts based on their metadata to get samples from a broad spectrum
    2. selected the 70 most-deployed fragments, which account for 90% of all assembly usage in our corpus.
    3. picked 100 more fragments from various clusters, to create a more representative and comprehensive collection of fragments,

主要结论

Summary of finding. Selected, representative findings of our study are:

  1. 23% of smart contracts in our corpus use inline assembly,
  2. contracts using inline assembly typically only include two fragments (2.08 on average) and 10.62 instructions,
  3. contracts that contain assembly are larger than those that do not (161 vs. 40 lines of code) and have more transactions,
  4. the percentage of contracts using inline assembly increases through time,
  5. 48 out of the 85 available instructions are used by less than 1% of the contracts that employ inline assembly,
  6. an inline assembly fragment typically uses instructions from at least three different instruction categories (e.g., arithmetic operations and system operations),
  7. inline assembly is mainly used for implementing functionality not available in Solidity as well as for gas optimization using specific code patterns.

RQ2: Smart Contract Characteristics：

The properties that we will utilize to examine the contracts are as follows: (1) number of transactions, (2) number of token transfers, (3) unique callers, (4) balances, (5) ERC token tag (i.e., fungible or Non-fungible Token (NFT)), (6) Etherscan labels, and (7) lines of code.

RQ3: Evolution of Inline Assembly

There is an evident trend that inline assembly usage in Solidity smart contracts increases over time. This result highlights the importance of inline assembly as a core component of smart contracts. Furthermore, there are some indications that developers use the latest compiler versions to compile contracts containing inline assembly while compiling other contracts with older versions.

RQ4: A Taxonomy of Inline Assembly

  1. Arithmetic Operations. 作为绝大多数内联汇编的基础运算，80.31% 的内联汇编合约都会用到。

  2. Comparison and Bitwise Logic Operations (77.91%). 算术操作，比如人为拓展512位运算、高精度小数平方根可能用到。

  3. Hashing Operations (0.97%). Only a few contracts use a hash function in inline assembly fragments. 非常少，很久之前 solidity 就内置了 sha3

  4. Environmental Information (80.86%). Chainid、extcodecopy 等属于这一类。

  5. Block Information (0.85%). 区块信息大多数源码层次足够了。

  6. Stack, Memory, and Storage Operations (90.94%). 最主要是内存相关操作。

  7. Flow Operations (24.17%). 主要是错误处理 switch，if 在复杂情况也会使用。jump 这一类的指令极少使用。

  8. System Operations (74.16%). 定义为 callback、revert、create 这几类。最多的是 delegatecall ，估计是代理合约。然后 create2 在 Solidity 不支持之前使用较为广泛。revert 是故意终止，return 正常执行后返回值。

  9. YUL Declarations. 主要是 let 定义临时使用的变量，极少有写内联汇编函数的，因为确实很笨拙。

  10. Logging Operations。一般用不到，不如直接 Solidity直接写方便。使用比例如下：

以上基本上是组合起来用：

  • 声明局部变量，用来暂存。YUL Declarations.
  • 内存读写，准备数据。Stack, Memory, and Storage Operations
  • 位操作进行细粒度计算。比较多是 eq，用来判断条件是否成立。Comparison and Bitwise Logic Operations
  • 算术操作贯穿始终。Arithmetic Operations.
  • 代理或者回调或者创建合约，也是非常常见的场景。System Operations
  • 对系统操作进行错误处理。Flow Operations

另外第二多的场景是：

  • memory 读取。
  • Bitwise operation，处理输出，比如将memory分成几部分。
  • 算术操作在内存偏移时很常用。

RQ5: USAGE OF INLINE ASSEMBLY

使用场景上分类：fragments that provide functionality not available in Solidity, fragments used to perform optimizations, and complementary fragments used to support another inline assembly fragment.

  1. Single Instruction Unavailable in Solidity. One common use of inline assembly is employing a single opcode instruction to obtain information not available through Solidity code. chainid extcodesize
  2. Deserialization. 处理内存数据后反序列化，比如为 ecrecover 做准备
  3. String, Bytes, and Math Utils.
  4. Proxy Implementation.
  5. Error Handling. 主要 switch revert
  6. Call other Contract.
  7. Optimizations for Cryptographic Primitives. 哈希和其他密码学相关
  8. Handling Arbitrary Data. 数据提取和处理
  9. Fine-Grained Memory and Storage Control. 比如strings库合约，提供字符串操作方法。
  10. Manual Inlining. A few fragments perform manual inlining using inline assembly, i.e., rather than using a library that implements certain functionality, they manually copy and paste only the assembly code into their code to reduce the gas costs. This class is mostly used for procedures from the ‘String, Bytes, and Math Utils’ class.

总结

对内联汇编的使用方式做了统计和分析，是不错的学习和研究内联汇编的入门级文章。静态分析工具应该支持内联汇编的分析，对于常用的库合约中的内联汇编，设计函数摘要。另外的用户编写的内联汇编应该设置最常见的错误提示。有些内联汇编的方式已经过时，Solidity 源码层面已经可以实现，也应该提醒，尽量减少内联汇编的使用，从而提高代码质量。

编译器优化和语言设计方面也应该做好，现在的优化还不太足够，根据使用场景和频次，对某些指令进行优化，也是很有意义的。
